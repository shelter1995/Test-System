# 陪练系统运行时一致性与场景持久化修复设计

## 目标

修复以下三个用户可见问题：

1. 陪练回复不得展示或保存模型的 `<think>` / `<thinking>` 思考内容。
2. 保存模型设置后，启动页立即显示 8003 当前生效的模型，同时实际陪练调用继续使用同一运行时配置。
3. 自定义场景创建后写入磁盘，在页面刷新和 8002 服务重启后仍可选择和使用。

## 设计

### 思考内容过滤

保留现有有状态流式过滤器，使跨 token 拆分的标签也能被识别。过滤发生在 SSE token 发出和 AI 消息写入会话之前，确保界面与历史记录一致。补充纯文本清理入口，供非流式兼容接口复用，避免旧调用路径重新暴露思考内容。

### 模型状态热刷新

8003 继续作为模型运行时配置的唯一事实来源。8002 的 `UnifiedLLMClient` 提供显式刷新模型信息的能力；`/api/status` 每次请求刷新一次并基于同一份快照返回 `ai_configured`、`ai_provider` 和 `ai_model`。模型设置页面保存成功后触发总览强制刷新。

实际对话请求不携带 8002 缓存的模型名称，仍转发至 8003 的 `/llm/chat` 或 `/llm/chat/stream`，由 8003 在请求时解析当前运行时端点。因此显示状态与实际模型通道共享同一来源。

### 自定义场景持久化

后端启动时从 `SCENARIOS_FILE` 读取场景，并只将标记为 `is_custom` 的合法记录合并到代码内置默认场景，避免旧文件覆盖新版默认配置。创建场景时先构造新记录，再可靠写盘；写盘失败必须返回错误，不能报告虚假成功。

前端初始化时调用 `/scenarios`，用后端返回的自定义场景重建下拉选项，同时保留内置选项和“自定义场景”入口。重复加载不得产生重复选项。

## 错误处理

- 思考标签未闭合时，标签后的内容视为思考内容，不输出也不保存。
- 模型状态刷新失败时，`/api/status` 返回未配置状态，不影响 8002 进程存活。
- 场景文件不存在或内容损坏时记录错误并仅使用默认场景。
- 场景保存失败时回滚内存新增项并返回 HTTP 500。

## 测试

- 流式标签跨 token、未闭合标签以及非流式文本清理测试。
- 模型信息缓存刷新测试，以及 `/api/status` 使用最新快照的测试。
- 自定义场景文件加载、损坏文件降级、保存失败回滚和接口返回测试。
- 前端场景恢复与模型设置保存后总览刷新回归脚本。
- 完成后运行 Tutor pytest、相关前端 Node 测试及 Python 编译检查。

## 非目标

- 不改变 8003 模型设置文件格式。
- 不新增数据库或用户级场景权限。
- 不重构默认场景定义和现有会话文件格式。
