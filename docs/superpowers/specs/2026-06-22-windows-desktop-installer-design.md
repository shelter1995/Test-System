# Test-System Windows 桌面安装包设计规格

**状态：** 待最终审阅

**日期：** 2026-06-22

**目标平台：** Windows 10/11 x64

**选定方案：** C#/.NET 8 WinForms 桌面宿主 + Microsoft Edge WebView2 + Inno Setup

## 1. 背景与目标

Test-System 当前由两个 Python 服务和一个浏览器前端组成：知识库服务默认监听 `127.0.0.1:8003`，AI 陪练与静态前端默认监听 `127.0.0.1:8002`。仓库已有 CPython 3.13.10 便携运行时构建能力，但最终用户仍需要解压并通过批处理启动，且后端控制台可见、浏览器关闭与后端退出没有可靠绑定。

本项目要增加面向非技术用户的 Windows 安装与桌面运行形态，同时保持现有源码开发流程不变。最终用户应能双击安装程序，选择程序目录和数据目录，完成安装并获得桌面快捷方式；双击快捷方式后只看到桌面应用窗口，关闭该窗口后本次启动的所有后端进程必须退出。

## 2. 范围

### 2.1 本期包含

- 生成单个 Windows x64 EXE 安装程序。
- 安装主体可完全离线完成，包括 CPython 3.13.10 x64、基础 Python 依赖、.NET 8 自包含桌面宿主和 WebView2 Evergreen Standalone Runtime。
- 安装向导允许分别指定程序安装目录和用户数据目录。
- 创建桌面快捷方式和开始菜单快捷方式。
- 使用无地址栏的 WebView2 桌面窗口承载现有前端。
- 静默启动、监控并停止两个 Python 后端。
- 支持现有生成文件的“另存为”下载。
- 安装结束时询问是否联网安装 MinerU 及 Pipeline 模型；跳过或失败不阻止基础版本使用。
- 支持覆盖升级、卸载和版本化发布。
- 为后续 agent 提供可自动验证的构建与验收契约。

### 2.2 本期不包含

- Windows 10/11 以外系统、32 位系统、ARM64 或 macOS/Linux 安装包。
- 将 MinerU 或模型放入主安装程序。
- 自动更新服务；新版本仍通过新的安装程序覆盖安装。
- 修改业务页面视觉设计。
- 将外部 LLM、Embedding 或 Rerank 服务离线化。
- 代码签名证书采购。构建应保留签名入口，但未配置证书时允许生成未签名测试包。

## 3. 不可破坏的兼容性约束

1. 原有源码开发模式继续有效：开发者仍可使用 `.venv` 和 `start_services.bat` 启动项目。
2. 安装包逻辑必须位于独立边界内：新增 `desktop-host/`、`installer/`，并只扩展现有 `packaging/`。
3. 对业务代码的数据路径改造必须采用“环境变量优先、当前路径回退”。没有安装版环境变量时，目录位置与当前行为一致。
4. 构建不得复制开发机 `.venv`、真实 `.env`、知识库、会话、日志、生成产物或模型缓存。
5. 所有构建输出写入 `dist-installer/` 或临时构建目录并由 Git 忽略。
6. 安装版的运行时、数据和下载文件之间不得使用构建机绝对路径。
7. 后续业务开发不需要启动或安装桌面宿主；只有桌面联调和发布构建需要 .NET 与 Inno Setup 工具链。

## 4. 总体架构

```text
桌面/开始菜单快捷方式
          |
          v
TestSystem.exe (.NET 8 x64, self-contained, WinForms)
  |-- 单实例锁与第二次启动激活
  |-- 读取安装配置和数据目录
  |-- 创建 Windows Job Object
  |-- 隐藏启动 RAG 服务 8003
  |-- 等待 /health
  |-- 隐藏启动 Tutor 服务 8002
  |-- 等待 /api/status
  |-- 打开 WebView2 -> http://127.0.0.1:8002
  |-- 处理下载、外部链接和错误页
  `-- 窗口关闭 -> 优雅退出 -> Job Object 兜底终止进程树

安装器 (Inno Setup)
  |-- 选择程序目录
  |-- 选择数据目录
  |-- 安装程序主体、CPython、基础依赖
  |-- 离线检测/安装 WebView2 Runtime
  |-- 写入安装配置并创建快捷方式
  `-- 可选启动 MinerU 在线增强安装器
```

桌面宿主是唯一面向用户的启动入口。现有批处理继续服务于源码开发和诊断，但安装版快捷方式不得指向批处理文件。

## 5. 目录与配置契约

### 5.1 程序目录

默认程序目录为 `{localappdata}\Programs\Test-System`，安装向导允许修改。程序目录只保存可由安装器覆盖或重建的文件：

```text
<InstallDir>/
├── TestSystem.exe
├── desktop-host runtime files
├── ai-tutor-system/
├── rag-anything-api/
├── packaging/
├── runtime/
│   ├── python/                 # CPython 3.13.10 x64
│   └── site-packages/          # 锁定的基础依赖
└── assets/
```

业务运行时不得写入该目录。安装器升级时允许覆盖这里的全部程序文件。

### 5.2 数据目录

默认数据目录为 `{localappdata}\Test-System\Data`，安装向导允许修改。数据目录必须是本机绝对路径；安装器要创建目录并验证当前用户具有创建、写入、重命名和删除临时文件的权限。

```text
<DataDir>/
├── config/
│   ├── install.json            # 安装 ID、数据目录和版本迁移状态
│   ├── rag.env
│   └── tutor.env
├── rag/
│   ├── storage/
│   └── output/
├── tutor/
│   ├── tutor_data/
│   └── generation_output/
├── runtime/
│   ├── optional-site-packages/ # MinerU 可选依赖
│   └── models/mineru/          # MinerU 模型与缓存
├── webview2/                   # WebView2 用户数据目录
├── logs/
│   ├── desktop-host.log
│   ├── rag.log
│   ├── tutor.log
│   └── mineru-install.log
└── temp/
```

安装器以机器可读配置文件保存数据目录，不依赖当前工作目录。桌面宿主读取配置后向子进程传递以下环境变量：

- `TEST_SYSTEM_DATA_DIR=<DataDir>`：总数据根目录。
- `TEST_SYSTEM_RAG_STORAGE_DIR=<DataDir>\rag\storage`。
- `TEST_SYSTEM_RAG_OUTPUT_DIR=<DataDir>\rag\output`。
- `TEST_SYSTEM_TUTOR_DATA_DIR=<DataDir>\tutor\tutor_data`。
- `TEST_SYSTEM_GENERATION_OUTPUT_DIR=<DataDir>\tutor\generation_output`。
- `TEST_SYSTEM_LOG_DIR=<DataDir>\logs`。
- `TEST_SYSTEM_OPTIONAL_SITE_PACKAGES=<DataDir>\runtime\optional-site-packages`。
- `TEST_SYSTEM_MODEL_DIR=<DataDir>\runtime\models\mineru`。

业务模块只读取与自身相关的变量。变量缺失时沿用现有仓库相对目录，确保源码模式兼容。

### 5.3 升级与数据目录保持

同一安装 ID 的覆盖升级必须沿用上次选择的数据目录，除非用户在安装向导中明确修改。修改数据目录时，安装器不自动搬迁旧数据，以免复制大型知识库造成不可控时间和磁盘占用；向导必须明确提示“只切换新版本使用的位置，旧数据仍保留在原目录”。数据迁移工具不属于本期范围。

## 6. 桌面宿主设计

### 6.1 技术形态

- .NET 8 WinForms x64，自包含发布，不依赖目标机器预装 .NET。
- `Microsoft.Web.WebView2` 固定版本，NuGet 锁文件纳入仓库。
- 发布为有窗口子系统的 `TestSystem.exe`，启动时不出现控制台。
- 宿主代码按职责拆分：配置读取、进程生命周期、健康检查、WebView2 窗口、下载处理和日志。

### 6.2 单实例

每个 Windows 用户只允许运行一个 Test-System 桌面实例。首实例持有命名互斥量；第二次启动不再创建后端，而是通知首实例恢复并置前窗口后退出。不得通过“端口已占用即复用”的方式连接未知服务。

### 6.3 后端启动

宿主使用安装目录内的 `runtime\python\python.exe`，精确版本必须为 3.13.10。启动前构造独立环境，不永久修改用户或系统 `PATH`、`PYTHONPATH`。

启动顺序：

1. 校验安装配置、Python、关键源码和基础依赖存在。
2. 校验数据目录可写并创建所需子目录。
3. 检查 `127.0.0.1:8003` 和 `127.0.0.1:8002`。若被非本实例占用，显示错误并终止启动，不杀死占用者。
4. 创建带 `KILL_ON_JOB_CLOSE` 限制的 Windows Job Object。
5. 使用隐藏窗口标志启动 RAG 服务，标准输出和错误重定向到滚动日志。
6. 轮询 `http://127.0.0.1:8003/health`，成功后启动 Tutor 服务。
7. 轮询 `http://127.0.0.1:8002/api/status`，成功后初始化 WebView2 并导航首页。

健康检查应有总超时、短连接超时和取消令牌。任何阶段失败都停止本次已创建的进程并显示中文错误页，错误页包含“重试”“打开日志目录”“退出”三个操作。

### 6.4 关闭语义

关闭主窗口等同于退出整个应用：

1. 禁止新请求并取消仍在等待的启动任务。
2. 向两个服务发送受本机随机令牌保护的内部关闭请求；关闭端点只接受回环地址，并要求请求头携带本次启动生成的令牌。
3. 最多等待 10 秒让两个服务完成当前写入并正常退出。
4. 对仍未退出的宿主直接子进程执行进程树终止。
5. 释放 Job Object，确保其余孙进程一并退出。
6. 刷新日志并退出桌面宿主。

Job Object 是崩溃、任务管理器结束宿主和正常关闭场景的最终兜底。宿主不得终止并非由它创建的同端口进程。

### 6.5 WebView2 行为

- WebView2 仅允许应用主视图导航到 `http://127.0.0.1:8002` 范围。
- `http`/`https` 外部链接使用系统默认浏览器打开；不能在应用主窗口覆盖本地应用。
- 禁用开发者工具、上下文菜单中的调试入口和不必要的浏览器快捷键；开发构建可通过显式开关启用调试。
- WebView2 用户数据固定到 `<DataDir>\webview2`，避免写入程序目录。
- 前端加载期间显示启动状态，不展示空白窗口。
- WebView2 Runtime 缺失或损坏时显示修复提示和安装日志位置，不自动联网下载。

### 6.6 文件下载

现有后端 `FileResponse` 和前端 Blob 下载均需继续工作。宿主订阅 WebView2 下载事件：

- 默认文件名取服务端 `Content-Disposition` 或 WebView2 建议文件名，并清理 Windows 非法字符。
- 每次下载显示系统“另存为”对话框，初始目录为用户的系统下载目录。
- 用户取消时取消 WebView2 下载，不生成半成品文件。
- 文件写入由 WebView2 执行，宿主不把整个文件加载到内存。
- 下载完成后显示轻量通知，可打开文件或所在目录。
- 同名文件是否覆盖由系统对话框确认。
- 下载目标不受应用数据目录限制；用户可以保存到任意有权限的位置。

为避免 `target=_blank` 产生不可控窗口，宿主处理新窗口事件：本地下载 URL在当前 WebView2 下载管线中执行，外部 URL 交给默认浏览器。

## 7. Python 运行时与依赖隔离

### 7.1 基础运行时

- 安装包固定 CPython `3.13.10` x64，构建时读取实际解释器版本和架构，任一不匹配立即失败。
- 基础依赖通过完整锁定文件安装到 `<InstallDir>\runtime\site-packages`。
- 构建必须从干净目标目录生成，不复制 `.venv`。
- 运行时使用 Python UTF-8 模式，禁止调用目标机器上的 `python`、`pip` 或 `uv`。
- 基础运行时为只读程序资产，升级时由安装器整体替换。

### 7.2 MinerU 可选增强

主安装包不包含 MinerU 包和模型。安装结束页提供默认不勾选的选项：“联网安装增强文档解析组件 MinerU 及 Pipeline 模型（体积较大）”。用户勾选后，由独立的图形化增强安装器在主安装完成后运行；主安装程序本身先成功结束，增强安装失败不得回滚主程序。

增强安装器契约：

1. 使用内置 CPython 3.13.10，安装固定 `mineru[core]==3.3.1` 及锁定依赖。
2. 安装目标先写入 `<DataDir>\runtime\optional-site-packages.installing`。
3. 不允许升级或覆盖 `<InstallDir>\runtime\site-packages`。
4. 安装后在隔离环境中验证 `mineru` 版本、CLI 可调用性和关键导入。
5. 调用 MinerU 3.3.1 提供的模型下载入口预取 Pipeline 模型，缓存路径由 `HF_HOME`、`HUGGINGFACE_HUB_CACHE`、`MODELSCOPE_CACHE` 和 `MINERU_TOOLS_CONFIG_JSON` 指向 `<DataDir>\runtime\models\mineru`。
6. 默认模型源选择适合当前网络环境的 ModelScope，并允许实现层在界面中切换 Hugging Face；模型源和实际命令写入日志。
7. 包与模型均验证成功后，以原子重命名替换正式 `optional-site-packages`，并写入状态清单。
8. 下载取消、断网、磁盘不足、导入失败或模型验证失败时删除临时依赖目录，保留可续传的模型缓存，基础应用仍可启动。
9. 应用内必须提供“安装/修复增强解析组件”入口，供首次跳过或失败的用户稍后重试。

MinerU 3.3.1 与 CPython 3.13.10 的实际 wheel/依赖兼容性是发布阻断项：构建或发布验证若不能在干净环境完成安装、导入和一次样例解析，不得发布带“可安装 MinerU”声明的版本。

## 8. 安装器设计

### 8.1 安装模式

使用 Inno Setup 生成单个 x64 EXE，默认采用当前用户安装，无需管理员权限。若用户选择受保护目录导致需要提升权限，安装器应在目录验证阶段明确提示重新选择可写目录；首版不切换为全机器安装，避免程序和用户数据权限语义混乱。

### 8.2 向导流程

1. 欢迎与系统要求检查：仅 Windows 10/11 x64。
2. 许可协议。
3. 选择程序目录。
4. 选择数据目录，并显示数据将包含知识库、配置、日志和模型。
5. 选择快捷方式，桌面快捷方式默认勾选。
6. 安装文件。
7. 静默检测 WebView2 Runtime；缺失或版本不满足时运行随包附带的 x64 Evergreen Standalone Installer。该步骤完全离线。
8. 写入安装配置、卸载信息和快捷方式。
9. 完成页提供“启动 Test-System”和“联网安装 MinerU 及 Pipeline 模型”两个选项。

### 8.3 升级

- 使用稳定不变的 Inno Setup `AppId` 识别同一产品。
- 新版本默认复用程序目录和数据目录。
- 升级前检测正在运行的 `TestSystem.exe`，提示用户关闭；安装器不得在后端仍写数据时覆盖文件。
- 采用先停止应用、替换程序文件、保留数据、启动新版本的顺序。
- 版本迁移失败时保留原数据和迁移日志；不得静默删除或重建知识库。
- 不支持安装较低版本覆盖较高版本，除非使用显式的开发/测试参数。

### 8.4 卸载

- 卸载程序、运行时、快捷方式和安装器注册信息。
- 默认保留数据目录。
- 卸载向导明确询问是否删除全部用户数据，并展示实际路径；只有用户主动勾选且路径通过安全校验时才递归删除。
- 若数据目录位于程序目录内部，仍将其视为用户数据，覆盖升级和默认卸载不得删除。
- 不删除用户通过下载对话框保存到其他位置的生成文件。

## 9. 版本、更新日志与发布流程

新增仓库根目录 `version.json` 作为人工维护的唯一产品版本源，采用语义化版本 `MAJOR.MINOR.PATCH`。构建过程从该文件派生：

- 桌面宿主程序集版本和文件版本；
- 安装器 `AppVersion` 与输出文件名；
- 运行时清单中的产品版本；
- 安装配置中的已安装版本。

`CHANGELOG.md` 继续作为面向维护者和用户的发布记录。每次正式发布必须：

1. 完成业务改动和自动测试。
2. 更新 `version.json`。
3. 将本版本变更写入 `CHANGELOG.md`，至少区分新增、修复、兼容性或安装变化。
4. 在干净构建目录生成运行时、桌面宿主和安装器。
5. 执行自动化产物检查。
6. 在 Windows 10/11 x64 干净环境完成安装验收。
7. 记录安装包 SHA-256；配置证书后再执行签名和签名验证。

建议输出名为 `Test-System-Setup-<version>-x64.exe`，不得使用会覆盖历史产物的无版本文件名作为归档件。

## 10. 日志、错误与安全

- 所有用户可见错误使用中文，包含可执行操作，不直接显示 Python 堆栈。
- 完整诊断写入数据目录日志；API Key、Authorization 头和敏感表单内容必须脱敏。
- 后端只监听 `127.0.0.1`，不得改为 `0.0.0.0`。
- 内部关闭接口若实现，必须要求每次启动生成的随机令牌，且仅接受回环地址请求。
- 下载文件名、配置路径和数据目录必须抵御路径穿越。
- 安装器递归删除前必须验证目标等于记录的数据目录、不是磁盘根目录、不是用户主目录、不是 Windows 或 Program Files 等受保护根目录。
- 进程日志采用大小或日期轮转，避免长期运行无限增长。
- 桌面宿主不得记录 API Key，不得把包含密钥的环境变量传给无关进程。

## 11. 测试策略

### 11.1 Python 自动测试

- 数据目录环境变量存在时，各业务目录全部落入指定根目录。
- 环境变量缺失时，现有源码目录行为保持不变。
- 生成产物列表与下载接口在外置目录下仍阻止路径穿越。
- 便携构建器固定 Python 3.13.10 x64并排除用户数据。
- MinerU 临时安装、验证、原子切换和失败回滚。

### 11.2 .NET 自动测试

- 安装配置解析和非法路径拒绝。
- 子进程环境构造不污染系统环境。
- 健康检查成功、超时、取消和后端提前退出。
- 端口占用时不启动、不误杀。
- 单实例第二次启动激活首窗口。
- 下载文件名清理、取消与目标路径选择。
- 窗口退出触发本次创建的进程树终止。

进程管理抽象应允许测试使用短生命周期的本地测试进程，不以 mock 调用次数代替真实退出行为。

### 11.3 构建产物检查

- 安装包、桌面 EXE、Python 和 WebView2 安装器均为 x64。
- Python 实际版本精确为 3.13.10。
- 安装内容不包含 `.venv`、真实 `.env`、用户数据和构建机绝对路径。
- `version.json`、桌面 EXE、运行时清单和安装器版本一致。
- 安装包包含 WebView2 离线安装器，不依赖联网完成主体安装。
- 构建在缺少锁定依赖、版本不一致或测试失败时返回非零退出码。

### 11.4 手工发布验收矩阵

- Windows 10 x64 和 Windows 11 x64。
- 无 WebView2 Runtime 的干净环境离线安装。
- 默认目录安装与自定义程序/数据目录安装，包括含空格和中文的路径。
- 桌面快捷方式启动、无控制台窗口、两个服务健康。

- 跳过 MinerU 后基础解析可用。

## 12. 验收标准

满足以下全部条件才视为完成：

1. 在 Windows 10/11 x64 上使用单个 EXE 完成离线主体安装。
2. 用户能选择程序目录和数据目录，安装后两者职责分离。
3. 桌面快捷方式启动 WebView2 应用，整个过程中无后端控制台窗口。
4. 关闭应用窗口后，本次宿主创建的所有后端进程在限定时间内退出。
5. 现有文件生成和下载功能可通过“另存为”保存到用户指定位置。
6. 未安装 MinerU 时基础功能不受影响；可选联网安装失败可恢复和重试。
7. 安装版固定使用 CPython 3.13.10 x64，基础依赖与 MinerU 依赖相互隔离。
8. 源码开发启动方式和默认数据路径行为保持兼容。
9. 覆盖升级保留数据，卸载默认保留数据。
10. 版本号、更新日志、构建和发布验证形成可重复流程。

## 13. 实施边界建议

后续实施计划应按以下顺序拆分，以便每一步都能独立测试：

1. 外置数据目录兼容层。
2. Python 运行时和依赖锁定增强。
3. .NET 桌面宿主的进程生命周期核心。
4. WebView2 窗口、导航与下载。
5. MinerU 图形化增强安装器。
6. Inno Setup 安装、升级和卸载。
7. 统一版本与一键发布构建。
8. 干净环境端到端验收。

实施不得把上述工作合并成一个不可测试的大脚本，也不得先写安装器再补数据目录和进程生命周期契约。
