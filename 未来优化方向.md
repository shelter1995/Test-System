# 未来优化方向

## RAG-Anything 检索性能优化

当前已经先完成主流程打通：陪练系统、方案编制、培训/出题流程默认接入 `http://localhost:8003` 的 RAG-Anything/LightRAG 服务。

后续可重点优化检索性能，尤其是陪练系统的实时响应速度。

### 现状判断

新检索链路比旧 RAG 慢，主要不是电脑配置问题，而是新旧链路工作量不同：

- 旧 RAG 主要是向量检索：问题 -> 向量库查片段 -> 返回结果。
- 新 RAG-Anything/LightRAG 链路会做知识图谱与向量混合检索，并调用大模型组织答案。
- 当前 `/ai_enhanced_search` 使用 `mix` 模式，适合生成高质量总结，但对实时陪练偏重。
- 陪练系统每轮会连续检索产品信息、销售话术、异议处理等内容，延迟会叠加。
- 新链路还依赖远程 API，包括 MiniMax LLM 和硅基流动 Embedding，网络/API 延迟会直接影响响应速度。

### 建议方向

1. 为陪练系统增加“快速检索接口”
   - 使用 `naive` 或 `local` 模式。
   - 或使用 `only_need_context=True`，只返回上下文片段，不让 LightRAG 每次都生成完整答案。
   - 陪练系统拿到片段后，再由自身 MiniMax 对话逻辑统一生成客户回复。

2. 区分实时场景和离线生成场景
   - 陪练系统：优先低延迟，使用快速上下文检索。
   - 方案编制、培训材料、测试题生成：可以继续使用 `mix` 模式，追求更完整的综合答案。

3. 降低 LightRAG 默认检索规模
   - 当前 `QueryParam` 默认 `top_k=40`、`chunk_top_k=20`，对陪练偏大。
   - 后续可按接口设置更小的 `top_k` 和 `chunk_top_k`。

4. 增加缓存
   - 对常见查询缓存结果，例如：
     - `商务视频彩铃 产品介绍 功能 价格`
     - `价格敏感型客户 应对话术 销售技巧`
     - `常见异议处理 价格异议 技术异议`
   - 陪练多轮对话中，同一产品和同一场景会反复查询，缓存收益较高。

5. 优化并发与超时策略
   - 陪练端当前已将 RAG 请求超时拉长到 90 秒，保证流程打通。
   - 后续可把三类检索并发执行，减少一轮对话总等待时间。
   - 同时应为超时结果保留清晰降级提示。

### 暂不处理事项

今天只做流程打通，暂不继续调整：

- 不重构 rag-anything-api 的检索接口。
- 不改变陪练系统的业务提示词结构。
- 不迁移旧 `rag-anything-api` 目录下的历史脚本。
- 不做性能压测和缓存实现。
