# 未来优化方向

## 已完成

- 主运行链路已统一到 RAG-Anything 服务层。
- 迁移导入脚本已统一复用 `RAGAnythingService`。
- 可执行 Python 路径已清理旧 LightRAG/旧向量/FAISS 直连栈。
- FastAPI 已从 `@app.on_event("startup")` 迁移到 lifespan 模式。
- 多库查询已改为 asyncio 并发执行，支持单库超时。
- 新增 `/context` 轻量上下文检索接口，陪练系统优先使用并自动回退。
- RAG 实例缓存已增加 LRU 上限和显式 unload 能力。
- 文档已清理旧存储路径/旧处理器说明。

## 后续优化

- 观察 `/context` 在实时陪练中的响应时间。
- 根据数据库数量调整 `MAX_RAG_INSTANCES` 和 `QUERY_ALL_TIMEOUT`。
- 如果 MinerU 入库耗时过高，再引入后台任务队列。
- 对常见查询增加缓存（同一产品/场景的反复查询收益较高）。
- 区分实时场景和离线生成场景：陪练优先低延迟，方案编制可继续用 mix 模式。
- 降低 LightRAG 默认检索规模（`top_k`、`chunk_top_k`）以适应实时场景。
